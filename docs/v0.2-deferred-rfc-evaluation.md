# v0.2 deferred RFC evaluation (H6 / M3 / M5)

> Per the v0.2 plan in `prompt_plan.md`, three findings from the
> v0.1.4 DA chain (H6 / M3 / M5) were judged **proportionality-deferred** —
> the cost of authoring them now exceeds the present marginal benefit, but
> they should be re-evaluated when specific triggers fire. This memo
> records the rationale, cost estimate, and trigger for each.
>
> Numeric thresholds are not duplicated here. They live in the external SSOT:
> `~/.claude/da-tools/thresholds.json` (`proportionality_thresholds` key).
> See `docs/proportionality-thresholds.md` for the link.

## H6 — self-check automation review

**Description**: A periodic / triggered review of the codebase's own
self-check infrastructure (CI gates, drift verifiers, contract tests).
Asks "is each self-check still load-bearing? Does it still pass for the
right reasons?"

**Cost**: ~2 hours per review iteration (read every gate, simulate failure
modes, confirm the failure surfaces correctly in CI).

**Benefit at v0.2**: Low. The verifier set is small (5 gates: ci.yml,
release.yml, sync-readme, verify-audit-schema, lint) and was just
authored; no drift expected for several iterations.

**Trigger to escalate**:
- A new external contributor lands a PR (signal: `external_contributor_pr_count`
  in SSOT exceeds 0)
- A quarterly review (every 3 months) coinciding with the
  `ai-role-assignment` matrix re-review (`__meta__.next_review`)
- A self-check fails for a reason that the author did not anticipate
  (signal: incident registered)

**Re-evaluation when**: The first of those triggers fires. If it has
been 6+ months without any of them, treat absence-of-signal as a signal
and run a one-off audit anyway.

## M3 — RFC 6-section template

**Description**: Standardize RFC documents (currently informal in
`docs/`) into a 6-section template (Context, Decision, Constraints,
Alternatives considered, Migration, Verification). Same structure as
mainstream OSS projects (Rust RFCs, Kubernetes KEPs, etc.).

**Cost**: ~1 hour to draft the template + ~30 min per existing/future
RFC to reformat. Cheap once authored, but value scales with reader
volume.

**Benefit at v0.2**: Low. Single-user repo with no external readers. The
existing free-form `docs/*.md` files already convey the same information.
Imposing a template is busywork without an audience.

**Trigger to escalate**:
- First external contributor submits an RFC PR
- The repo grows to multiple maintainers (signal: `contributor_count`)
- An RFC's rationale is misread or contested (signal: ambiguity in
  practice → need for shared structure)

**Re-evaluation when**: A new RFC is being written and the author
catches themselves wishing for a structure to follow. That is the
first true "would benefit" moment.

## M5 — release gate matrix doc

**Description**: A table cross-referencing release.yml's 6 conceptual
gates (versioning, changelog, CI, build, provenance, publish-approval)
against (a) which are active vs commented, (b) which are mandatory vs
opt-out, (c) what the failure mode and remediation is.

**Cost**: ~1 hour to author. Maintenance is "edit table when gate
changes" — proportional to gate churn.

**Benefit at v0.2**: Currently 4 of 6 gates active; the remaining 2
(provenance/SBOM, publish approval) are stubbed and conceptually
trivial. A 6-row table reads like an over-engineered version of the
existing `## Release procedure` section in CHANGELOG.

**Trigger to escalate**:
- Provenance/SBOM gate is implemented (RFC 0002 lands → 5 active gates,
  gate ordering becomes non-trivial)
- Publish-approval gate is implemented (RFC 0003 or equivalent → 6
  active gates, manual-vs-automated boundary needs documenting)
- An incident where a release shipped despite a gate that was
  silently disabled (signal: regression where the doc would have
  caught the misconfiguration)

**Re-evaluation when**: Either provenance or publish-approval gate is
about to be implemented. Author M5 as part of that PR, not separately.

## Coupling note

H6 and M5 share a property: both author **review tooling for the
review tooling**, and both are best deferred until the underlying
review infrastructure is large enough that "is it still working?"
becomes a real question. M3 is independent of those — its trigger is
audience scale, not infrastructure scale.

## Summary

| ID | Cost | Benefit @ v0.2 | Trigger |
|---:|------|----------------|---------|
| H6 | ~2h / iter | Low | First external PR / quarterly / unexpected gate failure |
| M3 | ~1h once + 30m / RFC | Low | First contested RFC / multi-maintainer |
| M5 | ~1h once | Low | Provenance or publish-approval gate implementation |

All three are **proportional** at their respective triggers and
**disproportional** before. This document is the receipt for that
judgment, so a future session can verify the trigger has actually
fired before reopening the work.
